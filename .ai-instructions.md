# Reglas para el Asistente AI

## Antes de cualquier cambio:
1. **SIEMPRE** revisar el estado actual de los archivos
2. **CONFIRMAR** los cambios específicos con el usuario
3. **NO** hacer múltiples ediciones seguidas sin confirmación
4. **PREGUNTAR** si hay dudas sobre el contexto
5. **NO eliminar ni modificar funcionalidades existentes que el usuario no haya solicitado cambiar.**

## Al implementar cambios:
- Hacer cambios graduales (un archivo a la vez)
- Esperar confirmación antes del siguiente paso
- No sobrescribir archivos sin verificar contenido actual

## Contexto y modo de operación
- Si el modo cambia entre "ask" y "agent", el asistente debe:
  - Preguntar al usuario si desea revisar el estado actual de los archivos y la conversación antes de ejecutar cualquier acción.
  - No ejecutar cambios automáticos sin confirmación tras un cambio de modo.
  - Mostrar un resumen del contexto actual antes de cualquier edición importante.

## NUEVA REGLA: Comentarios y documentación
- Cada función o método NUEVO o MODIFICADO debe incluir un comentario inicial (docstring) que describa claramente su propósito.
- Deben existir comentarios inline en bloques lógicos (inicio de captura, validaciones, manejo de errores, limpieza).
- Si una función carece de docstring, el asistente debe avisar y solicitar confirmación antes de modificarla.
- No eliminar comentarios existentes sin justificar y confirmar con el usuario.
- Los comentarios deben ser concisos, en español y orientados a mantenimiento futuro.

## REGLA PARA ELEMENTOS INTERACTIVOS EN FLUJOS DE AUTOMATIZACIÓN
1. Todo elemento interactivo que pueda aparecer más de una vez (inputs, selects, textareas, botones, etc.) debe ser recorrido con un ciclo `for` sobre la colección de elementos encontrados en el panel o sección correspondiente.
2. La búsqueda de elementos debe realizarse siempre dentro del contexto del panel actual (usando `panel.find_elements(...)` o `panel.find_element(...)`), nunca directamente sobre el `driver`, salvo que el elemento esté garantizado fuera de cualquier panel.
3. La lógica de interacción (clic, input, selección, etc.) debe estar encapsulada dentro del ciclo de recorrido de cada tipo de elemento, asegurando que todos los elementos presentes sean procesados.
4. Para nuevos elementos interactivos (como el botón "Continúa sin redes sociales"), se debe:
   - Buscar todos los elementos con el selector adecuado dentro del panel.
   - Recorrerlos con un ciclo `for`.
   - Validar su existencia en el DOM y que estén habilitados (`is_enabled()`).
   - No depender únicamente de la visibilidad (`is_displayed()`), para soportar ejecución headless.
   - Si el clic normal falla, intentar con JavaScript: `driver.execute_script("arguments[0].click();", elemento)`.
   - Registrar la acción en el log del flujo, indicando si fue clic normal o por JS.
5. La implementación de la interacción debe ubicarse en el bloque de procesamiento correspondiente al tipo de elemento, siguiendo el orden y estructura del flujo actual (inputs, selects, textareas, botones, etc.), y nunca fuera del ciclo principal de paneles.
6. Si existe duda sobre el contexto o la ubicación de un nuevo elemento, se debe consultar primero la estructura actual del flujo y confirmar con el usuario antes de proponer la implementación.

## Ejemplo de implementación para cualquier elemento interactivo:

```python
# Dentro del ciclo de paneles
elementos = panel.find_elements(By.XPATH, ".//TAG[contains(., 'Texto o atributo identificador')]")
for elemento in elementos:
    try:
        if elemento.is_enabled():
            try:
                elemento.click()
                TaskFillPortalForm.log_and_print("[Task] Elemento interactivo clickeado normalmente.")
            except Exception:
                driver.execute_script("arguments[0].click();", elemento)
                TaskFillPortalForm.log_and_print("[Task] Elemento interactivo clickeado por JS (no visible).")
    except Exception:
        TaskFillPortalForm.log_and_print("[Task] Elemento interactivo no se pudo procesar.")
```

Este ejemplo debe usarse como referencia para la implementación de cualquier nuevo elemento interactivo (inputs, selects, textareas, botones, etc.) en los flujos de automatización, siguiendo la regla definida arriba.

## REGLA CRÍTICA: Autorización explícita para modificar código

**NUNCA** modificar, editar o aplicar cambios en archivos de código sin una autorización explícita del usuario que contenga palabras como:
- "Aplica los cambios"
- "Procede con la modificación"
- "Actualiza el archivo"
- "Implementa el código"
- "Hazlo"

**PROHIBIDO:**
- Aplicar cambios automáticamente después de ejecutar tests o análisis
- Modificar código basándose en interpretaciones implícitas
- Asumir autorización por contexto conversacional
- **Decir que se aplicó un cambio cuando en realidad no se hizo (por estar en modo "ask" u otra limitación)**

**PROCESO OBLIGATORIO para cualquier cambio de código:**
1. Ejecutar test/análisis si se solicita
2. Reportar resultados y diagnóstico
3. Explicar la causa del problema
4. Mostrar SOLO el fragmento de código propuesto (usando bloques de código markdown)
5. **DETENERSE y esperar confirmación explícita**
6. Solo después de confirmación explícita: aplicar cambios

**NUNCA saltarse el paso 5.**

**Transparencia sobre limitaciones:**
- Si estoy en modo "ask" y no puedo editar archivos, **DEBO DECIRLO INMEDIATAMENTE** al usuario
- NUNCA decir que "se aplicó" o "se agregó" un cambio si no fue posible ejecutarlo
- Ser explícito sobre qué acciones puedo o no puedo realizar en el modo actual

**Ejemplo de lo que SÍ está permitido hacer sin confirmación:**
- Mostrar código en bloques markdown
- Explicar problemas
- Ejecutar tests/análisis cuando se solicita explícitamente
- Responder preguntas

**Ejemplo de lo que REQUIERE confirmación explícita:**
- Editar archivos `.py`, `.feature`, `.json`, `.yml`, etc.
- Crear nuevos archivos de código
- Modificar configuraciones del proyecto
- Aplicar correcciones de bugs

## REGLA CRÍTICA: Edición quirúrgica de archivos en modo agente

Cuando se solicite aplicar un cambio en un archivo existente:

1. **NUNCA** reescribir el archivo completo.
2. **NUNCA** modificar código fuera del bloque solicitado.
3. **NUNCA** reformatear, reordenar o alterar código adyacente al cambio.
4. **SIEMPRE** usar los marcadores `# ...existing code...` para representar código que NO debe tocarse.
5. **SIEMPRE** aplicar ÚNICAMENTE el cambio solicitado, nada más.
6. **SIEMPRE** leer el archivo antes de editar para ubicar el bloque exacto a modificar.
7. **SIEMPRE** esperar confirmación del usuario antes de aplicar cambios en archivos críticos como:
   - `task_fill_portal_form.py`
   - `test_steps_*.py`
   - Cualquier archivo de configuración (`.yml`, `.json`, `pyproject.toml`, etc.)

### Flujo de trabajo obligatorio con modo agente:
1. Leer el archivo actual para ubicar el bloque exacto
2. Mostrar el fragmento de código propuesto con contexto mínimo
3. Indicar exactamente en qué línea/bloque va el cambio
4. Esperar confirmación explícita del usuario
5. Aplicar ÚNICAMENTE ese fragmento sin tocar nada más
6. Confirmar al usuario qué se cambió y qué NO se tocó

### Archivos críticos que requieren confirmación explícita antes de editar:
- `src/tasks/task_fill_portal_form.py`
- `tests/test_steps_*.py`
- `src/questions/*.py`
- `features/*.feature`
- `conftest.py`
- `pyproject.toml`
- `docker-compose.yml`
- `.env`
